# 过程记录
选题源于在对测试开发的学习过程中反复出现的需求转用例十分耗时的问题，手工梳理一份中等复杂度的需求文档通常需要30分钟以上或者需要更多时间，且边界条件覆盖容易遗漏。使用这个工具后，初版用例生成时间压缩到5-10分钟，AI在异常场景和边界值测试上的覆盖面通常优于手工初稿，落地验证的效果符合预期。且能够直接生成excel格式的用例，方便一些同学在求助于ai生成初步测试用例后的格式问题。

在开发方式上，借助Kimi.ai和Trae的Builder模式将需求描述直接转化为项目骨架，再通过一次次的对话模式对具体模块进行迭代优化。这种方式我觉得不是传统先设计后编码的线性流程，从kimi的agent给出的大致项目框架和提示词就可以开始，到在TRAE中变成持续的对话式优化戴拿，开发重心从语法细节转移到系统架构和交互逻辑的设计判断上。

技术决策上，最初考虑前后端分离架构，但在Vibe Coding的快速验证思路下，我意识到对于7天测试周期，评审环境的无障碍运行比架构完整性更重要。因此果断精简为纯前端直调API方案，将部署成本降为零，确保开箱即用。

主要调优点：在完成初版并在本地实际录入“用户登录”等需求测试后，我深刻感受到大模型（DeepSeek-V3）在运用测试理论（等价类、边界值）时的逻辑严密性，生成的用例覆盖度极高。但这同时也暴露了一个致命问题：大模型生成十几条深度结构化的 JSON 文本耗时极长（约 1 分钟），在此期间页面处于“白屏死等”状态，极度影响交付体验因此使用流式输出。
初期发现AI返回格式存在波动——有时混入解释文字，有时字段名不一致。我通过两层手段解决：一是在Prompt中嵌入"必须返回JSON对象"的系统约束，并启用response_format: { type: 'json_object' }强制格式化；二是在前端增加字段映射的兜底逻辑，确保即使AI返回略偏标准结构，也能被正确解析。

这个学习过程中，我重新理解了Vibe Coding的本质：它并非替代思考，而是把开发者的开发或者测试的思想框架从如何写转移到需要什么和如何在生产过程中调优。当AI承担了实现层的工作后，提示词和验收判断的权重会在上升，这种转变让我对AI时代测试开发的工作边界有了新的认识。
